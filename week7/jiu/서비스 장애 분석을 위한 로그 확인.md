## 1. 서비스 상태 확인

```bash
systemctl status nginx
```

먼저 서비스가 현재 어떤 상태인지 본다.

- active인지 / failed인지
- 최근 exit code는 무엇인지
- 최근 에러 힌트는 있는지

## 2. 서비스 로그 확인

```bash
journalctl -u nginx.service -n 50
```

특정 서비스의 최근 로그를 본다.

다음과 같은 단서가 나올 수 있다.

- 설정 파일 문법 오류
- 포트 충돌
- 권한 문제
- 인증서 경로 문제
- 환경 변수/의존 서비스 문제

### 장애 시점 범위 좁히기

```bash
journalctl -u nginx.service --since "10 min ago"
```

장애 발생 전후 시점만 잘라서 로그를 본다.

### 에러 메시지와 실패 패턴 찾기

서비스 로그에서는 실패를 나타내는 공통 패턴을 먼저 찾는다.

대표적으로 많이 보는 키워드는 다음과 같다.

- `failed`
- `error`
- `denied`
- `timeout`
- `segfault`
- `killed`
- `oom`

예를 들어, 

- 에러 관련 로그만 조회
    
    ```bash
    journalctl -u nginx | grep -i error
    ```
    
- 최근 부팅에서 실패 로그만 조회
    
    ```bash
    journalctl -p err -b
    ```
    

### 시스템/커널 로그 확인

```bash
journalctl -p err -b
journalctl -k -b
```

서비스 로그만으로 원인이 보이지 않으면 **시스템 또는 커널 레벨 로그**도 함께 확인해야 한다.

서비스 자체의 문제가 아니라 다음과 같은 시스템 문제일 수 있기 때문이다.

- 메모리 부족
- 파일시스템 오류
- 커널 에러
- 디바이스 문제

특히 **OOM으로 프로세스가 강제 종료된 경우에는 커널 로그**에서 단서를 얻을 수 있다.

## 3.  `systemctl restart` 후 다시 로그 관측하기

## 4. 같은 패턴이 반복되면 로그를 읽고 원인을 좁히기

- `permission denied`
→ 파일 권한 또는 접근 권한 문제 가능성
- `address already in use`
→ 동일 포트를 다른 프로세스가 사용 중일 가능성
- `connection refused`
→ 백엔드 서비스가 실행되지 않았거나 연결 대상이 잘못되었을 가능성
- `out of memory`, `killed process`
→ 메모리 부족으로 프로세스가 종료되었을 가능성
- `failed to start`
→ 설정 오류, 실행 파일 문제, 의존 서비스 실패 가능성

# 실습: nginx 문법 오류를 발생시키고 로그로 원인 분석하기

## 실습 목표

- nginx 설정 파일에 문법 오류를 직접 발생시킨다.
- `systemctl status`와 `journalctl`로 장애 원인을 확인한다.
- syslog와 애플리케이션 로그에서도 같은 문제를 확인한다.
- 설정을 복구한 뒤 서비스를 정상적으로 재시작한다.

## 1. nginx 정상 상태 확인

먼저 nginx 서비스가 정상 상태인지 확인한다.

```bash
sudo systemctl status nginx
```

## 2. 설정 파일에 문법 오류 발생시키기

nginx 설정 파일을 연다.

```bash
sudo vim /etc/nginx/nginx.conf
```

이후 문법 오류를 넣어 nginx가 시작되지 않도록 만든다.

여기서는 세미콜론을 뺐다.

## 3. nginx 재시작 시도

문법 오류가 들어간 상태에서 nginx를 재시작한다.

```bash
sudo systemctl restart nginx
```

설정 파일 오류가 있어서 재시작이 실패한다.

## 4. 서비스 상태 확인

서비스가 어떤 상태로 실패했는지 확인한다.

```bash
systemctl status nginx.service
```

여기서 확인할 수 있는 내용은 다음과 같다.

- 상태 (`failed`)
- `exit-code`
- 최근 에러 메시지

## 5. `journalctl`로 로그 확인

### 상세 로그 확인

```bash
journalctl -xeu nginx.service
```

- `x` : 로그 메시지에 대한 추가 설명 제공
- `e` : 로그의 끝부분으로 바로 이동
- `u nginx.service` : nginx 서비스 로그만 조회

최근 nginx 서비스 로그를 바로 확인하면서 실패 원인을 좁혀 볼 수 있다.

### **최근 시점 로그만 확인**

```bash
journalctl -u nginx.service --since "10 min ago" -n 10
```

### 실패 패턴만 확인

```bash
journalctl -u nginx | grep -i failed
```

## 6. 로그 분석

실제 로그는 다음과 같다.

```csharp
Apr 23 19:10:39 localhost.localdomain systemd[1]: Starting nginx.service - The nginx HTTP and reverse proxy server...
Apr 23 19:10:39 localhost.localdomain nginx[2777]: nginx: [emerg] invalid number of arguments in "root" directive in /etc/nginx/nginx.conf:44
Apr 23 19:10:39 localhost.localdomain nginx[2777]: nginx: configuration file /etc/nginx/nginx.conf test failed
Apr 23 19:10:39 localhost.localdomain systemd[1]: nginx.service: Control process exited, code=exited, status=1/FAILURE
Apr 23 19:10:39 localhost.localdomain systemd[1]: nginx.service: Failed with result 'exit-code'.
Apr 23 19:10:39 localhost.localdomain systemd[1]: Failed to start nginx.service - The nginx HTTP and reverse proxy server.
```

`Apr 23 19:10:39 localhost.localdomain nginx[2777]: nginx: [emerg] invalid number of arguments in "root" directive in /etc/nginx/nginx.conf:44`

- `root` 지시어의 인자 개수가 잘못되었고
- 문제 파일은 `/etc/nginx/nginx.conf`
- 문제 위치는 `44번째 줄`

→ 이 장애의 직접 원인은 **nginx 설정 파일의 문법 오류**임을 확인할 수 있다.

## 7. 복구

문법 오류를 수정한 뒤 다시 저장하고, nginx를 재시작한다.

## 8. syslog에서도 확인 가능

같은 장애는 syslog 파일에서도 확인할 수 있다.

```bash
sudo grep -i nginx /var/log/messages | tail -n 20
```

### 전체 로그 흐름에서 보기

```bash
sudo tail -n 50 /var/log/messages
```

## 9. 애플리케이션 로그에서도 확인 가능

nginx 자체 에러는 애플리케이션 로그에서도 확인할 수 있다.

```
sudo tail-n50 /var/log/nginx/error.log
```

이 로그는 nginx가 직접 남긴 에러 로그이므로, 설정 파일 오류나 런타임 오류를 더 직접적으로 보여줄 수 있다.

> 
> 
> 
> ### 같은 장애라도 확인 위치가 다르다.
> 
> - `journalctl` → 서비스 실행/실패 맥락
> - `/var/log/messages` → 시스템 전체 로그 흐름
> - `/var/log/nginx/error.log` → nginx 자체 에러 상세
