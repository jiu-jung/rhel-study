# Bare Metal, VM, Container

## 핵심 비교

| 구분 | Bare Metal | VM | Container |
| --- | --- | --- | --- |
| 기본 구조 | 물리 서버에 OS와 앱 직접 설치 | Hypervisor 위에 게스트 OS 실행 | 호스트 OS 위에서 격리된 프로세스 실행 |
| 격리 단위 | 물리 서버 | VM | 컨테이너 |
| OS 사용 방식 | 서버가 OS 직접 사용 | VM마다 게스트 OS 보유 | 호스트 OS 커널 공유 |
| 실행 단위 | 서버 | 게스트 OS + 앱 | 앱 프로세스 |
| 성능 | 오버헤드 가장 적음 | Hypervisor 오버헤드 존재 | VM보다 가벼움 |
| 배포 속도 | 느림 | 중간 | 빠름 |
| 자원 활용률 | 낮아지기 쉬움 | 서버 통합 가능 | 여러 앱을 가볍게 실행 |
| 이식성 | 하드웨어/OS 의존 큼 | VM 이미지 단위 이동 | 컨테이너 이미지 단위 이동 |
| 운영 단위 | 서버 | VM | 컨테이너, Pod, 이미지 |
| 대표 사례 | 고성능 DB, HPC, 전용 서버 | 레거시 앱, 테스트 환경, DR | 마이크로서비스, CI/CD, 클라우드 네이티브 앱 |

## 한 줄 정리

- **Bare Metal**
    - 물리 서버를 직접 사용하는 방식
    - 성능은 좋지만 배포·확장 유연성 낮음
- **VM**
    - 물리 서버 위에 여러 OS를 올리는 방식
    - 격리성은 좋지만 게스트 OS 때문에 무거움
- **Container**
    - 호스트 OS 커널을 공유하면서 앱을 격리 실행하는 방식
    - 가볍고 빠르며 배포 이식성이 좋음

# Container 기본 구조

## 컨테이너란

- 애플리케이션을 격리된 환경에서 실행하는 단위
- 별도 OS를 실행하는 것이 아님
- 호스트 OS 커널을 공유
- Linux kernel 기능으로 실행 환경 격리
    - namespace
    - cgroup
    - mount
    - network
    - user
    - PID

## 실행 흐름

```
Container Image
    ↓ run
Container
    ↓ start
Isolated Process
```

## 핵심 개념

| 개념 | 설명 |
| --- | --- |
| Image | 애플리케이션 실행에 필요한 파일, 라이브러리, 설정을 담은 패키지 |
| Container | 이미지를 실행한 인스턴스 |
| Runtime | 컨테이너 프로세스를 실제로 생성하는 구성 요소 |
| Registry | 이미지를 저장하고 배포하는 저장소 |
| Volume | 컨테이너 외부에 데이터를 저장하기 위한 연결 지점 |
| Network | 컨테이너 간 또는 외부와 통신하기 위한 네트워크 구성 |

# Container Image

## 이미지의 역할

- 애플리케이션 실행 환경 패키징
- 코드, 라이브러리, 설정 파일 포함
- 동일한 이미지를 여러 환경에서 실행 가능
- 개발·테스트·운영 환경 차이를 줄이는 데 사용

## 이미지 레이어 구조

- 이미지는 여러 개의 읽기 전용 레이어로 구성
- 각 명령 또는 변경 사항이 레이어로 쌓임
- 컨테이너 실행 시 쓰기 가능한 레이어가 추가됨

```
Read-only image layers
    +
Writable container layer
    =
Running container filesystem
```

## 이미지 기반 실행의 장점

- 빠른 배포
- 동일한 실행 환경 재현 가능
- 버전 관리 용이
- 롤백 용이
- CI/CD 파이프라인과 결합하기 쉬움

# Container Lifecycle

## 생명주기

```
image
  ↓ run/create
created container
  ↓ start
running container
  ↓ stop
stopped container
  ↓ rm
deleted
```

## 주요 명령 흐름

```bash
podman pull nginx
podman run -d --name web nginx
podman ps
podman stop web
podman rm web
podman rmi nginx
```

## 상태별 의미

| 상태 | 의미 |
| --- | --- |
| Image | 실행 가능한 컨테이너 패키지 |
| Created | 컨테이너는 생성됐지만 아직 실행 전 |
| Running | 컨테이너의 메인 프로세스 실행 중 |
| Stopped | 프로세스 종료, 컨테이너 메타데이터는 남아 있음 |
| Deleted | 컨테이너 제거 완료 |

# Container 내부 프로세스

## PID 1

- 컨테이너 내부의 첫 번째 프로세스
- 컨테이너의 메인 프로세스
- 이 프로세스가 종료되면 컨테이너도 종료됨

## 예시

```bash
root@container:/# ps -aux

USER   PID  COMMAND
root     1  nginx: master process nginx -g daemon off;
nginx   29  nginx: worker process
nginx   30  nginx: worker process
root   199  /bin/bash
```

## 해석

- PID 1
    - `nginx: master process`
    - 컨테이너의 메인 프로세스
- PID 29, 30
    - Nginx worker process
- `/bin/bash`
    - `podman exec` 또는 `docker exec`로 접속한 셸 프로세스

# Container 격리 구조

## namespace

- 프로세스가 보는 시스템 자원을 분리
- 컨테이너별 독립된 실행 환경 제공

| namespace | 격리 대상 |
| --- | --- |
| PID namespace | 프로세스 ID 공간 |
| Network namespace | 네트워크 인터페이스, IP, 라우팅 테이블 |
| Mount namespace | 파일시스템 마운트 지점 |
| UTS namespace | hostname, domain name |
| IPC namespace | 프로세스 간 통신 자원 |
| User namespace | 사용자 ID, 그룹 ID |

## cgroup

- 프로세스의 리소스 사용량 제어
- CPU, memory, block I/O 등 제한 가능

| 기능 | 설명 |
| --- | --- |
| CPU 제한 | 특정 컨테이너의 CPU 사용량 제한 |
| Memory 제한 | 컨테이너 메모리 사용량 제한 |
| I/O 제한 | 디스크 I/O 사용량 제어 |
| 프로세스 관리 | 리소스 사용량 추적 |
